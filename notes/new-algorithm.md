# 新算法：面向高代价蛋白质设计工作流的自适应工具链规划算法

## 一句话定义

该算法不是“从几个蛋白质工具里检索一个工具”，而是：

**在高代价、长链路、可失败、可恢复的蛋白质设计工作流中，动态生成、评估、裁剪并修正工具链，使系统以更低成本获得更高任务成功率。**

它本质上是一个**工作流级动态规划器**，而不是一个简单的 ToolKG 检索器。

---

## 1. 为什么需要这个算法

蛋白质设计领域的可用工具数量本身并不大，因此如果问题被表述为“基于 ToolKG 找到合适工具”，很容易退化成：

- 简单遍历
- 规则筛选
- 静态模板匹配

这类问题很难支撑“核心算法”。

真正困难的地方不在“找工具”，而在：

- 如何把少量工具组合成**可执行的多阶段工具链**
- 如何在预算、风险和目标约束下选择更值得尝试的链路
- 如何在中间结果变差时决定继续、止损、patch 还是 replan
- 如何减少一次错误高代价调用带来的后续浪费

因此，该算法研究的对象是：

**高代价蛋白质设计工作流的工具链规划与运行时修正问题。**

---

## 2. 更明确的问题定义

### 2.1 任务对象

给定一个蛋白质设计任务：

- 自然语言目标，例如稳定性、功能性、结合性质等
- 结构化约束，例如长度范围、模板、预算、工具白名单/黑名单
- 当前执行上下文，例如已完成步骤、失败历史、质量门禁结果、剩余预算

系统需要输出的不是单个工具，而是：

- 一条当前最值得执行的工具链
- 或对现有工具链的局部修正 `patch`
- 或对后续工具链的整体替换 `replan`

### 2.2 核心优化目标

该算法要联合优化以下目标：

- 最终任务成功率尽可能高
- 总执行成本尽可能低
- 无效高代价工具调用尽可能少
- 恢复代价尽可能低
- 人工介入次数尽可能少且时机更合理

### 2.3 与普通工具检索的区别

普通工具检索关注：

- 哪个工具最匹配当前请求

该算法关注：

- 哪条**多步工具链**更可能完成任务
- 哪条链路更便宜、更稳健、更易恢复
- 中途失败后是否值得继续投入

所以它不是 `tool retrieval`，而是：

**constraint-aware + budget-aware + risk-aware + recovery-aware planning**

## 3. 形式化定义

可以把该算法定义为一个带运行时反馈的工具链规划问题。

### 3.1 输入

输入记为：

- 任务目标 `g`
- 约束集合 `c`
- 工具集合及其能力图 `K`
- 当前执行上下文 `x_t`
- 当前运行时观测 `o_t`

其中：

- `g` 描述蛋白质设计目标
- `c` 包含长度、模板、成本预算、风险约束、允许工具等
- `K` 是小规模但结构化的工具能力图
- `x_t` 是当前执行到第 `t` 步的上下文
- `o_t` 是当前获得的运行时观测，例如质量指标、失败码、分歧信号

### 3.2 输出

输出记为：

- 当前候选工具链集合 `Pi_t = {pi_1, pi_2, ..., pi_k}`
- 默认推荐链 `pi*`
- 或针对当前链路的局部修补 `patch`
- 或针对后缀的重规划 `replan`

### 3.3 目标函数

直观上，算法要最大化：

`任务收益 - 执行成本 - 风险代价 - 恢复复杂度`

更具体地说，任一候选链 `pi` 都可以被赋予一个综合价值：

`Score(pi) = alpha * Feasibility + beta * GoalFit - gamma * Cost - delta * Risk - eta * RecoveryComplexity`

这里不要求论文里一定写成严格数学优化，但要明确：

- 可执行性是硬约束
- 成本、风险、恢复复杂度是联合决策变量
- 排序不是单目标，而是多目标权衡

---

## 4. 算法的核心组成

### 4.1 候选工具链生成

算法不会直接确定一条链，而是先生成若干条可执行候选链。

候选链通常遵循蛋白质设计工作流中的阶段结构，例如：

`序列生成 -> 结构预测 -> 质量门禁 -> 结构条件优化 -> 再预测 -> 目标打分`

这一步要解决的不是“有哪些工具”，而是：

- 工具链是否满足 I/O 闭包
- 阶段顺序是否合理
- 当前约束下哪些链路合法

### 4.2 候选链多目标评分

对每条候选链计算综合分值。评分至少应考虑：

- `Feasibility`：当前是否可执行
- `GoalFit`：与设计目标的匹配程度
- `Cost`：预计时间、算力、外部调用开销
- `Risk`：失败或低质量结果的概率
- `RecoveryComplexity`：后续若失败，修复是否困难

这一层决定：

- 哪些链可以被保留
- 哪条链作为默认推荐
- 哪些链应提前淘汰

### 4.3 预算感知候选裁剪

当任务预算有限时，并不是所有候选链都值得继续深入。

因此算法需要根据当前状态决定：

- 保留几条候选链
- 是否需要展开更深的后续步骤
- 是否应提前做廉价验证再决定是否继续高代价步骤

这一步体现“自适应”的第一层含义：

**搜索规模会随任务难度、预算和风险动态变化。**

### 4.4 运行时自适应修正

执行过程中，一旦得到新的中间观测，算法要重新判断当前链路是否仍值得执行。

典型运行时观测包括：

- 结构预测置信度
- QC 失败码和失败原因
- 多候选之间的一致性或分歧
- 最近一次 patch 是否有效
- 已消耗预算与剩余预算

在这些信号基础上，算法动态选择：

- `continue`：继续执行当前链
- `patch`：局部调参或替换某一步
- `replan`：保留前缀，重做后缀
- `stop`：提前止损

### 4.5 恢复链集成

该算法不是把失败视为异常，而是把失败视为规划过程中的正常组成部分。

因此恢复流程应被纳入算法闭环：

`retry -> patch -> replan`

也就是说：

- `retry` 是最小恢复
- `patch` 是局部结构修补
- `replan` 是策略级调整

---

## 5. “自适应”到底指什么

这里的自适应不是笼统的“更智能”，而是三个明确的动态调整机制。

### 5.1 候选数量自适应

- 简单任务：少量候选即可
- 复杂任务：保留更多候选，避免过早收缩

### 5.2 搜索深度自适应

- 当前链路表现稳定：继续推进
- 当前链路信号变差：提前验证、缩小搜索或止损

### 5.3 恢复范围自适应

- 局部问题：优先 patch
- 结构性问题：优先 replan

所以该算法也可以被理解为：

**一种带预算感知、风险感知和恢复感知的工具链规划算法。**

---

## 6. belief-state 在这里处于什么位置

### 6.1 为什么需要它

如果算法要做到真正的自适应，就不能只依赖静态规则。

因为很多关键量都不是直接可观测的，例如：

- 当前链路继续成功的概率
- 当前失败是偶发噪声还是结构性困难
- 当前剩余恢复空间是否还足够
- 继续投入一次高代价预测是否值得

这些都需要通过运行时信号进行估计。

### 6.2 在该算法中的角色

在本算法中，`belief-state` 更适合被理解为：

**内部的轻量状态估计模块**

而不是单独的主算法标题。

它可以维护如下估计量：

- `p_success`：当前链路继续成功的概率
- `p_structural_failure`：当前失败属于结构性失败的概率
- `recovery_margin`：剩余可恢复空间
- `expected_remaining_cost`：完成当前链路的剩余成本
- `intervention_value`：此时人工介入的边际价值

### 6.3 它如何发挥作用

这些状态估计将影响：

- 候选链排序
- 候选链裁剪
- patch/replan 决策
- 是否提前止损

因此更准确的关系是：

- 主算法：自适应工具链规划算法
- 内部模块：轻量 belief-state 估计

不是：

- 整个课题都等同于完整的 belief-state controller

## 7. 可直接写进论文的算法定义

可以采用下面这段定义：

> 面向高代价蛋白质设计工作流的自适应工具链规划算法，是一种工作流级动态规划方法。该算法将蛋白质设计任务建模为一个带成本、风险和恢复约束的多阶段规划问题，而非单步工具检索问题。算法首先基于任务约束、工具能力图和工作流阶段结构生成若干可执行候选工具链，再通过可执行性、目标匹配度、预估成本、预估风险和恢复复杂度进行多目标排序；在执行过程中，算法结合结构预测质量、门禁结果、失败历史和预算消耗等运行时观测，维护对当前链路成功概率、结构性失败风险和剩余恢复空间的估计，并据此动态决定继续执行、局部 patch、后缀 replan 或提前止损，从而减少高代价无效调用并提高整体任务成功率。

## 8. 与现有项目架构的对应关系

该算法与当前项目设计是相容的。

它可以自然映射到已有架构中的几个位置：

- `Tool Retrieval / Candidate Generation / Candidate Scoring / Selection`
- `Patch / Replan`
- `HITL` 决策点
- `StepResult / Safety / EventLog` 提供的运行时观测

因此它不是要推翻原系统，而是在原系统上把“如何规划和修正工具链”提升为一个明确算法层。

## 9. 该算法依靠了哪些研究思路

### 9.1 工具使用与工具链生成

相关工作：

- ToolLLM
- Gorilla

启发：

- 这些工作说明“工具检索”成为核心问题，通常依赖超大工具空间和复杂 API 文档。
- 这反过来说明，在蛋白质设计这种小工具空间里，不能把贡献简单表述为检索。
- 因此应将问题上升为“多阶段工具链规划”。

### 9.2 测试时预算分配与路由

相关工作：

- BEST-Route
- EcoAct

启发：

- 不是所有任务都值得投入相同的计算预算。
- 工具或动作不应被无差别展开。
- 需要根据当前难度和价值决定是否扩大候选、继续执行高代价步骤或提前收缩。

### 9.3 步骤级搜索与价值引导

相关工作：

- QLASS
- LATS

启发：

- 长链路任务不能只看最终输出。
- 中间步骤的价值评估会显著影响整体决策质量。
- 因此工具链规划应利用中间观测重新评估当前链路的长期价值。

### 9.4 不确定性感知决策

相关工作：

- UoT

启发：

- 不确定性应进入决策闭环，而不是只作为描述信息。
- 这为轻量 belief-state 的引入提供了直接理论依据。

### 9.5 多 Agent 失败分析与恢复治理

相关工作：

- Why Do Multi-Agent LLM Systems Fail?
- Which Agent Causes Task Failures and When?

启发：

- 多 Agent 失败常来自协作链、验证不足和失败归因困难。
- 因此 `patch/replan` 不应被视为事后补丁，而应成为算法本身的一部分。

## 10. 该算法为什么仍然可以成为核心算法

即便蛋白质设计工具数量少，这个算法仍可能成立，原因在于：

- 难点不在单步选工具，而在多步链路规划
- 难点不在检索规模，而在高代价工作流中的动态权衡
- 难点不在静态模板，而在运行时修正与恢复决策

因此，它的贡献不应写成：

- “提出了一个更好的 ToolKG 检索方法”

而应写成：

- “提出了一种面向高代价蛋白质设计工作流的自适应工具链规划方法”

## 11. 可观察的验证指标

为了体现该算法作用，指标不应局限于“工具选对率”，而应采用工作流级指标：

- Top-1 计划最终成功率
- 平均执行成本
- 平均执行时间
- 平均高代价步骤调用次数
- 平均 patch 次数
- 平均 replan 次数
- 人工介入次数
- 失败后的恢复效率

这些指标才能真正证明：

**算法的价值不在于把某个工具选对，而在于让整个高代价工作流更少走错路。**

## 12. 该算法尚需回答的六个算法前置定义

上文已经给出了该算法的总体定义，但若要让它真正成立为“核心算法”，还必须先回答六个前置建模问题。

这六个问题不是实现细节，而是算法是否自洽的基础。如果这些问题不被明确，整个方法就很容易退化成：

- 几个启发式分数的堆叠
- 若干人工阈值的门控系统
- 事后解释性的经验规则

因此，这一部分可以被视为该算法的“建模地基”。

### 12.1 Cost Schema：如何定义和确认代价

首先必须明确，所谓“高代价”不能只停留在口头描述，而必须被拆解为可操作的代价结构。

在当前语境下，代价至少应包括：

- `compute_cost`：计算资源消耗，例如 GPU 时间、推理时长、远程调用次数
- `time_cost`：整体任务推进所消耗的时间
- `opportunity_cost`：预算被当前链路占用后，其他候选无法展开的损失
- `recovery_cost`：当前动作失败后引入额外 patch/replan 的代价
- `human_cost`：触发人工确认或人工介入的额外开销

这里最关键的问题不是“有哪些代价”，而是：

**这些代价如何被确认，而不是凭主观感觉指定。**

一个较稳妥的做法是采用三层来源：

- `prior_cost`：工具级先验成本，例如根据工具类型预设低/中/高
- `historical_cost`：根据历史日志统计平均耗时、失败后额外代价
- `online_cost`：在当前任务执行中根据真实耗时和调用情况动态修正

因此，本算法中的成本不应被写成单一常数，而更适合写成：

`Cost = f(prior_cost, historical_cost, online_cost)`

若做降阶版本，也至少要在论文中说明：

- 成本由哪些分量构成
- 成本如何归一化
- 成本阈值如何确定

否则“成本感知”会变成空话。

### 12.2 Risk Schema：如何定义风险，而不把它混同于成本

风险必须与成本显式区分。

原因在于：

- 高成本动作不一定高风险
- 低成本动作也不一定低风险

例如一次结构预测可能昂贵，但结果稳定；而某次廉价 patch 虽然便宜，却可能把系统推入后续更难恢复的路径。

因此，风险应至少反映以下内容：

- 当前链路失败的概率
- 当前链路产出低质量结果的概率
- 当前动作导致后续结构性失败的概率
- 当前动作扩大风险暴露的可能性

风险的信号来源可以包括：

- 候选链内部的一致性或分歧程度
- 工具历史稳定性
- 当前步骤的不确定性指标
- QC 拒绝原因与失败码
- 多次 patch/replan 后的路径恶化迹象

这里的关键点是：

**风险不应等于失败次数，也不应等于成本分数。**

它是对“未来不利结果可能性”的单独估计变量。

### 12.3 Recovery Schema：如何定义恢复空间与恢复复杂度

对于该算法而言，仅考虑当前一步是否成功是不够的，还必须考虑：

- 当前失败后还能否恢复
- 恢复的成本高不高
- 恢复之后原有有效前缀还能保留多少

因此必须单独定义恢复相关变量。

可以至少包含：

- `retry_budget`：当前还允许多少次重试
- `patch_feasibility`：当前问题是否仍可通过局部 patch 修复
- `suffix_replan_preservability`：当前已完成前缀中有多少可以保留
- `recovery_complexity`：若当前链失败，未来恢复路径大概会有多长
- `human_rescue_value`：人工介入是否能显著降低恢复难度

这部分非常重要，因为它直接决定系统是否会做出“局部看起来省钱、长期却更昂贵”的错误决策。

换句话说：

**恢复空间是该算法区别于普通静态排序方法的关键。**

### 12.4 State Schema：哪些变量属于隐状态

这一部分可以直接借鉴 belief-state 的讨论。

本算法若要真正自适应，就不能只看表面观测，而必须明确哪些核心量属于“不可直接观测、需要估计的内部状态”。

典型隐状态可包括：

- `task_difficulty`：当前任务真实难度
- `evidence_sufficiency`：当前证据是否足够支撑继续执行
- `toolchain_stability`：当前链路是否处于稳定区间
- `structural_failure_tendency`：当前失败是否更像结构性失败而非偶发噪声
- `recovery_margin`：剩余恢复空间
- `intervention_value`：当前请求人工介入是否值得

关键在于：

- 这些变量不能被直接观测
- 它们又决定了动作选择

因此它们必须被明确建模为状态，而不是混杂在 feature 列表里。

如果这一步不做，整个算法就会退化成：

- 读几个指标
- 算一个总分
- 超阈值就换动作

这会明显削弱算法性。

### 12.5 Observation Schema：如何定义可观测信号及其到状态更新的映射

一旦状态是隐的，就必须回答另一个问题：

**系统到底能看到什么？这些观测如何影响状态估计？**

可观测信号至少包括：

- 结构预测输出及其置信度指标
- QC/gate 的通过、失败和失败码
- 候选链路之间的一致性与分歧
- 历史 patch/replan 的成败轨迹
- 已消耗的时间、预算和高代价调用次数
- Safety 触发信息
- 人工确认与人工修改记录

难点在于，这些观测是高度异构的：

- 有的是连续数值
- 有的是离散标签
- 有的是事件序列
- 有的是人类决策痕迹

因此必须给出至少一种明确说明：

- 哪类观测主要影响哪类状态
- 观测冲突时如何处理
- 哪类观测可靠度更高
- 何时应显著更新 belief，何时应保持保守

这一部分本质上就是简化版的 `belief update mechanism`。

即便不采用完整概率模型，也应至少给出：

- 显式更新规则
- 分层加权更新
- 或事件触发式状态修正机制

### 12.6 Action-Utility Schema：如何定义动作价值与决策准则

最后，算法不能只停留在“给状态和观测打标签”，它必须回答：

**在当前状态估计下，不同动作哪个更值得执行？**

因此必须明确动作空间以及动作价值函数。

动作空间可以包括：

- `continue`
- `expand_candidates`
- `shrink_candidates`
- `patch_local`
- `suffix_replan`
- `full_replan`
- `request_hitl`
- `stop`

但光有动作列表还不够，还要定义动作效用考虑哪些因素，例如：

- 当前动作对成功率的提升
- 当前动作带来的即时成本
- 当前动作引起的未来恢复代价
- 当前动作对剩余预算的消耗
- 当前动作对人工介入机会的影响

因此，动作价值不应只看“眼前最省”，而应是对未来若干步收益与代价的近似估计。

换句话说：

**该算法不是单步贪心的工具链排序器，而是近似面向长期效用的工作流决策器。**

---

## 13. 这六个前置定义之间的关系

这六个定义不是相互孤立的，而是组成一个闭环：

1. `Cost Schema` 说明动作和链路要付出什么代价
2. `Risk Schema` 说明未来坏结果的可能性
3. `Recovery Schema` 说明失败之后还有多少挽回空间
4. `State Schema` 说明真正需要估计的隐变量是什么
5. `Observation Schema` 说明如何用运行时证据更新这些状态
6. `Action-Utility Schema` 说明在当前状态估计下如何选动作

因此，该算法真正的核心并不是“生成工具链”这一动作本身，而是：

**在不完整观测下，对成本、风险和恢复空间进行联合建模，并据此做工作流级动作选择。**
