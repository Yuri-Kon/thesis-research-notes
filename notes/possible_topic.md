# 可能的突破点

## 目前的判断

如果这套系统继续按照 “蛋白质助手+门控+HITL+难度评分” 推进的话，很容易被视为另一个 “生命科学助手” 的子集。

核心原因在于研究对象没有变，而不是功能少的问题：

> 用户提出生命科学任务 -> 多Agent分工 -> 调用工具 -> 给出设计/建议

无论中间添加多少 `gate, HITL, replan, pathc` 本质上都是在跑 “助手系统的执行增强版”，这类改动不足以构成一个毕设课题。

近两年的多Agent文献，越来越不关注 “集成多个Agent”，而是放在：

- 如何定义任务难度
- 如何控制协调成本
- 如何处理多 Agent 特有失联
- 如何做 benchmark, rubric, formal control

## 当前的方向

如果继续做该课题，只会有两个结果：

1. 继续做 “生命科学助手的一个更强的runtime”
   - 这里很容易论文父集模块
   - 如何论证该系统不是对方系统的一个执行器
1. 把课题对象改成 “科学工作流控制/测评/编译问题”
   - 这里生物领域只是实验场景，不再是系统定义本身

所以应当把研究对象从 “生命科学设计助手” 切换到 “高代价科学工作流的控制与评测算法”。这是突破口。如果做不到就要另行寻找。

## 可能的突破点

1. 科学工作流难度建模与预算化控制

   这里的核心是：

   - 给定一个 workflow graph
   - 估计每步的失败概率、成本、信息增益、不确定性
   - 决定是否继续自动执行、是否切换工具、是否提早 HITL、是否终止

   这里对应的应该是： `constrained decision making / scheduling / control under uncertainty`

   可以做成一个底层控制器：

   - workflow 图模型
   - 难度状态预估器
   - constrained controller

1. 科学 Agent Workflow Benchmark

   比如面向高成本科学任务的多智能体工作流 benchmark 与自动评测方法

   这里的 benchmark 不是 生命科学助手的子集，而是独立的研究资产，因此可以去做任务难度分层、failure taxonomy、自动 rubric、代价/成功率/可追踪性联合评测等内容，但可能缺少核心算法。

## 进一步考虑算法

### 难度控制器

它的本质应该是：`面向工作流的约束序贯决策算法`

其结构应该如下：

- 输入：任务工作流、当前执行状态、预算、失败记录、不确定性
- 输出：下一步控制动作
- 目标：在成本、风险、人工干预预算的情况下，最大化成功率或收益

可以被表述为：

- Constrained Sequential Decision Making(约束序贯决策)

  序贯决策问题的一类扩展。核心在于除了优化主要目标（比如收益最大化）之外，还需要满足一定的约束条件。约束可以是资源限制、安全约束、风险上限等。

- Contextual Bandit(上下文赌博机)

  强化学习中的一种简化序贯决策问题。Contextual Bandit不考虑动作对未来状态的长期影响。每一步的决策只关注当前“上下文”（context）下选择哪个动作获得最优即时奖励。

- CMDP(约束马尔可夫决策过程)

  这是Constrained Sequential Decision Making在MDP框架下的具体实现。它结合了MDP的状态-动作-转移结构和约束条件。

  既考虑长期累积奖励，又保证约束不被违反，是工业和安全关键场景中常用模型。

- Risk-aware Scheduling(风险感知调度)

  这是序贯决策在调度领域的特殊应用，决策不仅考虑效率或收益，还需要控制风险或不确定性。

  在调度过程中，每一步动作都可能带来风险（如延迟、资源冲突、任务失败），策略需要在期望收益和风险之间权衡。

- MDP(马尔可夫决策过程)

  描述序贯决策问题的经典数学模型。它的核心是“状态-动作-转移-奖励”的框架，用于建模一个智能体在环境中逐步做出决策的过程。

如果从算法角度考虑，创新点在于：

1. 难度是隐变量，不是静态标签
   难度要由任务结构、工具可靠性、历史失败、信息不确定性共同推断
1. 难度必须决定策略
   难度高时，不是终止，而是改变探索深度、候选规模、是否进入HITL、是否提前终止、是否切换工具
1. 难度要在线更新
   同一个任务，执行中途探测到失败或高置信证据后，策略要重算

如果从这个角度出发，算法的核心就可以被概括为：

> 状态预估 + 风险/成本建模 + 预算约束下的策略选择

可以把题目设置为 “高代价科学工作流的通用控制算法”，从蛋白质设计中独立出来。

### 自动benchmark

算法核心分成：

1. 任务表示与生成
   把科学任务表示为结构化的 task graph / DSL / rubric tree, 然后自动合成不同难度的实例
1. 难度校准
   通过一些方式去估计 hardness
1. 自动评分
   看轨迹是否有效、代价是否合理、错误是否可恢复、是否满足约束等条件

所以其核心算法可以总结为：

> 任务生成 + 难度校准 + 轨迹级评分

但是其中最关键的是 “难度校准”，不过会很清楚的问题：

- 如果自动生成不够强，容易被质疑 “只是整理测试题”
- 如果自动判分不可靠，学术性会被削弱
