# 面向高代价蛋白质设计工作流的自适应工具链规划算法：待讨论问题清单

## 文件目的

本文件不再重复算法定义，而是整理：

- 从**进行研究**的角度，当前还需要明确什么
- 从**得出结论**的角度，最终需要回答什么
- 从**着手实现**的角度，落地前需要拍板什么

它的作用是帮助后续继续收敛问题边界，避免算法定义已经成立，但研究目标、实验目标和实现范围仍然发散。

---

## 一、研究层面仍需讨论的问题

### 1. 主问题到底是什么

当前算法涉及多个可能的优化方向：

- 降低高代价调用浪费
- 提高最终任务成功率
- 提高工具链规划质量
- 改善 patch/replan 决策质量
- 减少人工介入次数

这些目标不能在研究叙述中并列为“同等主目标”。

必须明确：

- 哪一个是**主问题**
- 哪些只是次级收益

否则后续论文会失去主线。

### 2. 核心研究假设是什么

算法成立依赖若干前提，但这些前提目前还没有被明确写成研究假设。

例如：

- 高代价步骤确实会主导整体工作流成本
- 中间运行时观测对后续成功率具有预测价值
- 当前恢复链长度可以近似反映未来恢复代价
- 轻量 belief-state 比静态阈值更能表达真实任务状态

这些都需要明确，否则算法只是“看起来合理”。

### 3. 课题边界到哪里为止

需要进一步确认：

- 算法只覆盖 `Planner + runtime adaptation` 吗
- 是否把 `HITL` 触发也纳入动作空间
- 是否包含 `Safety` 信息融合
- 是否包含人工修改后的再规划逻辑

如果边界不清，系统中的每一层都可能被吸进来，工作量会迅速失控。

### 4. 与 belief-state 的关系如何最终表述

目前较稳妥的表述是：

- 主算法：自适应工具链规划算法
- belief-state：内部轻量状态估计模块

但仍要进一步决定：

- 是否在题目或章节标题中突出 belief-state
- 是否把算法命名为 `belief-state guided planner`
- 是否只在方法部分说明其是内部状态表示

这是一个很关键的叙事问题，会直接影响毕设是否看起来偏向 controller。

### 5. 最终创新点到底写哪些

不能把所有部分都写成创新点。

更合理的做法是提前压缩成 2 到 3 个核心创新，例如：

- 将高代价蛋白质设计工作流形式化为带恢复与预算约束的工具链规划问题
- 提出带轻量 belief-state 的运行时工具链修正机制
- 提出恢复感知的多目标候选排序与动作选择方法

这一步必须提前做，否则后续会出现“内容很多但创新点不集中”的问题。

### 6. 研究风险主要是什么

需要提前确认至少三类风险：

- 风险 1：算法退化为规则系统
- 风险 2：观测信号不足，belief-state 没有明显增益
- 风险 3：工具空间太小，无法体现规划算法价值

每类风险都需要对应缓解方案。

## 二、结论层面仍需讨论的问题

### 1. 最终希望证明什么命题

算法最终必须服务于一个清晰的结论，而不是泛泛地说“表现更好”。

至少要在以下命题中选一个作为主命题：

- 在相近成功率下，显著减少高代价步骤调用
- 在相近总成本下，提高任务成功率
- 相比静态方法，更少进入长恢复链
- 相比简单门控，更合理地分配 patch / replan / stop

不同命题会直接决定实验设计。

### 2. 哪些 tradeoff 是可接受的

现实中很可能不会出现“所有指标都更好”的情况。

因此必须提前讨论：

- 如果成功率提高，但平均时间更长，是否可接受
- 如果成本降低，但人工介入稍增，是否可接受
- 如果恢复链变短，但部分任务更早止损，是否可接受

这决定你最后如何解释实验结果。

### 3. 如果出现负结果，如何解释

必须提前想好：

- 如果 belief-state 版本没有显著优于简单动态版本，说明什么
- 如果动态规划只在高难任务上有效，是否仍成立
- 如果成本下降但成功率也下降，是否说明策略过于保守

这部分不能等实验结束后再临时解释。

### 4. 结果分析的粒度应该多细

除了总体平均指标，还要提前决定：

- 是否要做任务分层分析（简单/复杂）
- 是否要做失败类型分析
- 是否要做 patch/replan 触发前后的案例分析
- 是否要做人工介入样本分析

如果只做平均值，算法的具体作用机制往往看不见。

### 5. 基线和对照会支持怎样的结论

不同基线回答的问题不同：

- 静态模板链：回答动态规划是否比静态计划更优
- 固定阈值门控：回答 belief-state 是否优于简单规则
- 不带 belief-state 的动态版本：回答状态估计是否真有额外价值

因此需要先决定希望结论落在哪一层对比上。

## 三、实现层面仍需讨论的问题

### 1. Lite 版本的最小范围是什么

最关键的问题不是“完整算法能做什么”，而是“最小可运行版本包含什么”。

建议优先只保留：

- 候选链生成
- 多目标评分
- 轻量 belief-state 更新
- `patch / replan / stop` 三类关键动作

其余动作可暂时不实现。

### 2. 成本 schema 如何落地

需要尽快形成一张最初版的成本表，例如：

- 哪些工具是高代价步骤
- 高/中/低成本如何映射
- 时间成本如何记录
- 恢复成本如何近似

否则“cost-aware”无法进入代码与实验。

### 3. 风险 schema 如何落地

需要确定：

- 哪些观测作为风险信号
- 风险是否做成分层标签还是连续值
- 风险与成本如何同时进入评分

否则算法很容易把风险重新折叠回成本。

### 4. belief-state 的状态变量具体保留哪些

不应一开始就纳入过多状态。

建议先讨论最小状态集，例如：

- `p_success`
- `p_structural_failure`
- `recovery_margin`
- `expected_remaining_cost`

如果变量过多，后续更新规则和实验解释都会变得很混乱。

### 5. 观测从哪里来

实现前必须明确：

- 哪些观测直接来自 `StepResult`
- 哪些来自 `Safety`
- 哪些来自日志聚合
- 哪些需要额外统计

否则方法写得再好，最后也没有可靠输入。

### 6. 状态更新机制做成什么形式

需要拍板：

- 显式规则更新
- 分层加权更新
- 事件触发式更新
- 简单学习式映射

在当前时间条件下，更现实的通常是显式且可解释的更新方式。

### 7. 动作触发边界如何设定

必须明确：

- 什么情况下用 `patch`
- 什么情况下直接 `replan`
- 什么情况下提前 `stop`
- 什么情况下请求 `HITL`

这一步如果不提前定义，实现时会不断摇摆。

### 8. 基线如何实现

实现前需要确定至少三类基线：

- 静态 Top-1 链路
- 固定阈值门控
- 不带 belief-state 的动态版本

否则即使算法实现成功，也很难通过实验说明其价值。

### 9. 实验任务集如何组织

还需明确：

- 任务从哪里来
- 是否按难度分层
- 每类任务数量大概多少
- 是否需要控制预算或工具可用性变量

任务集组织方式会直接影响结论可信度。

### 10. 代码实现顺序如何安排

实现不应从“最复杂版本”开始。

更合理的顺序可能是：

1. 静态候选生成与评分
2. 成本/风险 schema 接入
3. 轻量 belief-state 状态定义
4. 运行时更新与 `patch/replan` 触发
5. 基线对照实验

这有助于控制风险。

---

## 四、当前最值得优先讨论的八个关键问题

若要继续收敛，我认为优先级最高的是以下八个问题：

1. 该算法的**唯一主目标**是什么
2. 轻量 belief-state 中**到底保留哪些状态变量**
3. 成本如何做成**可确认的量化 schema**
4. 哪些运行时观测是真正可靠的
5. `patch` 和 `replan` 的触发边界如何定义
6. 最小可实现版本包含哪些动作
7. 实验基线最终选哪些
8. 结果最希望支撑哪一个主结论

这八个问题如果不先定下来，后续无论是继续研究还是开始编码，都很容易失焦。

---

## 五、建议的下一步讨论顺序

为了让后续工作真正推进，建议按下面顺序继续讨论：

1. 先确定**主问题与主结论**
2. 再确定**最小状态集与最小动作集**
3. 然后确定**成本/风险/恢复 schema**
4. 再确定**基线与实验任务集**
5. 最后才进入实现与验证

这样可以避免过早写代码，却在后面发现研究问题本身没有收敛。
