\section{候选方向比较}

在完成选题目标的重新定义之后，接下来的问题便是：在多个候选研究方向中，哪一条最适合在当前时间、已有系统基础和答辩要求下收敛成一个成立的毕业设计课题。从目前的讨论来看，真正有代表性的路线大致可以分为四类：继续沿生命科学助手增强路线推进、转向自动基准测试（benchmark）、转向工作流编译/验证/修复，以及转向运行时控制器（controller）。它们分别对应不同的研究对象、不同的创新重心，也对应不同的风险。

\subsection{延续“生命科学助手增强”路线}

这是与当前系统最接近的一条路线。其基本思路是继续围绕生命科学任务本身展开，将系统定义为一个面向蛋白设计或更广义生命科学任务的多智能体助手，并在此基础上不断补充能力，例如更完整的工具链、更复杂的任务拆解、更细致的门控、更丰富的人工介入交互，以及更完善的恢复逻辑。

这一方向的核心本质是：\textbf{把系统当作应用平台来完善。} 它关注的是系统覆盖了多少能力、能处理多少真实任务、交互链路是否顺畅以及执行流程是否稳定。它的优势很明显：与现有系统最贴近，实现成本最低，也最容易展示“系统已经做出来了”。

但它的问题同样明显。首先，这条路线最容易被看成另一个生命科学助手的自然扩展，因为只要系统的对外叙事仍然是“面向生命科学任务的多智能体帮助系统”，那么内部无论加入多少状态机、恢复机制和人工介入逻辑，本质上都没有脱离“助手系统”这一概念。其次，这一路线的创新点通常很难集中，容易分散在多个工程增强点上。最后，在当前时间压力下，继续走“大系统增强”路线很容易导致功能越做越多，主线却越来越模糊。

因此，这条路线虽然工程上最平滑，但从毕设选题的角度看恰恰风险最高，因为它最容易落入“系统很复杂，但核心研究问题不清晰”的局面。

\subsection{自动基准测试路线}

自动基准测试路线的思路，是把研究对象从“如何更好地完成任务”转向“如何系统地评测复杂科学工作流任务”。在这一方向上，系统不再首先关注“输出什么结果”，而是关注：如何构造具有不同复杂度和难度层级的任务，如何让这些任务可复现、可比较、可自动评分，以及如何让 benchmark 真正反映多智能体系统在高代价科学任务中的能力差异。

从近两年的研究趋势来看，这条路线是成立的，而且学术价值较明确。PaperBench\parencite{paperbench} 强调的是研究复现实验的细粒度自动评分与 rubric 拆解，BrowseComp\parencite{browsecomp} 强调的是高难度信息寻求任务中的持久搜索，CoLLAB\parencite{collab} 则突出协作复杂度和多智能体 benchmark 的可扩展构造。

这一路线的本质是：\textbf{把“难度”本身变成研究对象。} 它不像助手系统那样关注“给出什么答案”，而是关注“什么样的任务更难、为什么难、如何被自动构造出来、又如何被可靠评分”。从独立性上看，这条路线很安全，因为 benchmark 不是生命科学助手的子模块，反而可以把不同系统都当作被评测对象。

不过，这条路线也有明显问题。首先，它的重点不在控制算法，而在任务生成、难度分层和自动评分机制上。如果自动生成和自动评分不够强，benchmark 很容易退化成“整理了一批测试集”。其次，这条路线虽然能很好地体现“难度”，但与当前系统的直接耦合不如 controller 紧。最后，在剩余时间内，如果要同时做好任务构造、难度校准和轨迹级评分，工作量并不小。

因此，自动基准测试是一条学术边界很清晰、不容易成为父集子集的路线，但它更像“独立评测平台”选题，和当前系统的天然结合度略低。

\subsection{工作流编译、验证与修复路线}

另一条可能的方向，是把研究对象定义为“从科研目标到可执行工作流的编译、验证与修复”。在这条路线下，重点不再是任务本身，也不是最终结果，而是：如何从自然语言目标生成结构化工作流；如何检查工具链的 \texttt{I/O} 兼容性、参数合法性和资源约束；以及如何在执行失败后进行最小修复和后缀重规划。

从方法论上看，这条路线非常有计算机专业特色，因为它天然带有 \texttt{compiler}、\texttt{verification}、\texttt{schema checking} 和 \texttt{plan repair} 的色彩。现有多智能体系统设计中，越来越多工作开始强调结构化任务表示、明确的状态机组织以及对工具链和交互协议的形式约束。例如 MetaAgent\parencite{metaagent} 这类工作已经开始从显式状态机组织的角度构造多智能体系统。

这条路线的核心本质是：\textbf{把科学任务的执行链看作一种可编译、可验证、可修复的程序结构。} 从理论表达上看，它很有吸引力，也确实比纯助手系统更像计算机问题。

但在当前语境下，这条路线有两个现实问题。第一，它与现有系统已有能力重合较多。当前系统已经具备显式状态机、计划候选、\texttt{patch/replan}、候选验证和恢复机制，因此如果再把“工作流编译与修复”作为核心方向，很容易被视为对现有 \texttt{planner} 和 runtime 的进一步完善，而不是一个新的研究主线。第二，这条路线虽然形式化色彩较强，但在应用价值的叙述上相对弱一些。它更像“让流程更正确”，而不是“让系统在高代价条件下做出更合理的运行时决策”。

因此，工作流编译/验证/修复路线是一个理论气质较强的候选方向，但与当前系统的差异化空间并没有想象中大。

\subsection{运行时控制器路线}

controller 路线的出发点，是把研究对象上升为：在高代价、长链路、带恢复和人工介入的科学工作流中，系统究竟应如何决定下一步如何推进。在这一方向上，系统不再被理解为一个“完成生命科学任务的助手”，而被理解为一个“治理工作流执行的运行时控制器”。

从近两年的研究来看，这一方向虽然还没有与本课题完全同构的现成答案，但已经有多条重要脉络构成了理论背景。在不确定性建模方面，UoT 说明不确定性可以进入 agent 的规划闭环\parencite{uot}；在推理资源方面，BEST-Route 说明可以依据任务难度和质量阈值动态调度推理计算资源\parencite{best-route}；在中间决策方面，QLASS 说明 stepwise value 可以进入 agent 推理过程\parencite{qlass}；在动作经济性方面，EcoAct 说明工具和动作选择可以受到成本约束\parencite{ecoact}；在多智能体系统层面，Why Do Multi-Agent LLM Systems Fail? 和 Which Agent Causes Task Failures and When? 则说明系统失败很大程度上与协作、验证和治理机制相关\parencite{whyfail,whichandwhen}。

这一路线的本质是：\textbf{把难度、风险、工作量和人工介入价值统一纳入一个运行时决策问题。} 它所决定的不是“下一句如何回答”，而是诸如：
\begin{itemize}
	\item 当前是否值得继续执行；
	\item 是否应扩大候选探索；
	\item 是否应切换到更保守的路径；
	\item 当前失败更适合局部修复还是后缀重规划；
	\item 是否应请求人工介入；
	\item 是否应提前终止以避免进一步浪费成本。
\end{itemize}

controller 路线的优势在于，它与当前系统契合度最高。因为现有系统已经具备状态机、人工介入、恢复链、阶段化执行链和实验框架，控制器可以自然嵌入这些已有能力之上，而不需要推翻架构。同时，它也是最容易把“难度”从口头概念变成真正算法变量的方向。

当然，这条路线最大的风险在于，如果最终只做成“几个特征、若干阈值、超过阈值就人工介入”的规则系统，那么创新性会明显不足。因此，controller 方向真正成立的前提是：它必须被明确建模为一个部分可观测、带风险与预算约束的序贯决策问题，而不是一个更复杂的工程门控模块。

\subsection{不同方向的比较与选择理由}

综合来看，这几个方向的差异可以概括如下：生命科学助手增强路线最贴近现有系统，但独立性最弱，也最容易被已有父集覆盖；自动基准测试路线独立性最强，难度问题也最自然，但与当前系统直接结合稍弱；工作流编译/验证/修复路线理论上较“计算机化”，但与现有平台能力重叠较多；而 controller 路线则位于一个相对平衡的位置，它既能显著提升选题层级，使课题从“助手系统”转向“工作流治理”，又能最大限度复用现有系统作为实验平台，同时还能把“难度、风险、工作量、恢复和人工介入”这些分散概念统一到一个核心决策问题中。

因此，如果在当前时间节点下需要选择一条既能体现算法性、又能形成明确主线、同时还能落到现有系统的路线，那么 controller 是最值得优先考虑的方向。而在 controller 内部之所以进一步倾向于 \texttt{Belief-State Controller}，正是因为它最适合把“难度”定义为一个运行时隐状态，而不是一个静态标签或单纯打分结果。
