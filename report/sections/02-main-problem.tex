\section{当前选题面临的核心问题}

在已有系统基础上，如果毕业设计继续沿原有思路推进，其主要问题并不在于系统功能是否足够丰富，而在于选题主线尚未完成从应用系统到研究问题的提升。这会使当前课题在创新性、独立性和答辩可解释性方面都面临明显风险。

\subsection{缺少一个足够明确的核心算法问题}

目前系统已经包含多智能体协作、显式状态机、人工介入、恢复链以及蛋白设计工作流等多个组成部分，但这些组成部分更多体现为一个较完整的执行平台，而不是一个被单独抽象、建模和论证的算法对象。

换句话说，当前系统已经回答了“如何把流程跑起来”的问题，却还没有清晰回答“系统依据什么原则做关键治理决策”。尤其在以下方面，仍缺少统一建模：
\begin{itemize}
	\item 如何刻画任务难度；
	\item 如何估计执行风险；
	\item 如何衡量工作量与代价；
	\item 如何在不同风险和预算条件下调整执行策略；
	\item 如何决定是否继续自动执行、请求人工介入或直接进入恢复流程。
\end{itemize}

虽然系统中已经存在若干门控和打分机制，但它们目前更接近启发式聚合和静态排序，而不是一个能够支撑论文主线的核心算法。因此，如果继续沿当前路径推进，课题很容易被理解为“一个做得较完整的系统”，却难以被概括为“一个明确的算法问题”。

\subsection{当前选题仍然容易被归入“生命科学助手”父集}

另一个更现实的问题是，当前系统的对外叙事仍然可以被概括为：

\[
\texttt{生命科学任务输入}
\rightarrow
\texttt{多智能体规划与执行}
\rightarrow
\texttt{工具调用}
\rightarrow
\texttt{结果输出}
\]

只要这一叙事不改变，那么无论系统内部加入多少状态机、人工介入和恢复逻辑，它仍然很容易被看作“生命科学助手”的一种更复杂实现。也就是说，它可能被视为：
\begin{itemize}
	\item 生命科学助手的执行引擎；
	\item 生命科学助手的后端工作流；
	\item 或生命科学助手的工程增强版本。
\end{itemize}

这会直接带来答辩风险：系统虽然复杂，但很难说清楚它与已有生命科学助手系统之间的本质区别，也难以解释为什么它不是已有方向的子模块。

\subsection{现有增强机制更偏工程健壮性，而非算法创新}

当前系统已经拥有许多合理且重要的机制，例如显式状态机、人工介入、恢复链、快照和事件日志等。这些机制很好地保证了执行一致性、恢复能力和系统治理意识，也使系统在工程上具备较高完成度。

但从毕业设计的角度看，这些机制更多体现的是系统设计的正确性和工程健壮性，而不是一个统一的算法原理。换句话说，当前系统“强在平台”，但“弱在算法主线”。如果不能进一步回答“系统根据什么状态和依据做出关键治理动作”，那么这些机制最终仍然会被看成合理的工程增强，而非独立的研究贡献。

\subsection{难度、风险和工作量尚未进入统一决策闭环}

此前反复出现的关键词包括难度、风险、工作量、不确定性、成本和人工介入时机。这些词都指向同一个根本问题：系统真正缺少的不是更多功能，而是一个能够统一解释并驱动关键治理动作的决策框架。

如果没有这一框架，那么系统就无法真正解释：
\begin{itemize}
	\item 为什么某个任务更难；
	\item 为什么某个阶段更值得请求人工介入；
	\item 为什么某条恢复路径优于另一条；
	\item 为什么此时应该继续执行、切换策略或直接止损。
\end{itemize}

也就是说，当前系统已经接近一个可用的工作流平台，但还缺少“控制依据”这一层核心算法。

\paragraph{小结}
因此，当前选题面临的核心问题可以概括为三点：第一，系统已具备较完善的平台能力，但尚未抽象出一个足够独立的核心算法问题；第二，只要主叙事仍停留在“生命科学多智能体助手”，就很难避免被已有父集系统覆盖；第三，在当前时间约束下，必须尽快将研究对象从“继续扩展系统功能”转向“提炼一个可论证、可验证、可答辩的核心问题”。
