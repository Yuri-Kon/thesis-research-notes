\section{Belief-State Controller 作为研究主线的理论依据与问题定义}

\subsection{从当前系统到控制问题的提出}

在前文的分析中，已经可以看到当前系统并不缺少工作流执行机制。无论是显式有限自动机、阶段化蛋白质设计流程、还是\texttt{HITL}、\texttt{patch/replan}与事件审计，这些能力都表明系统已经具备较完整的多Agent runtime结构。换言之，当前系统所面临的主要矛盾，已经不再是“能不能把系统跑起来”，而是“在一个高代价、长链路、可失败、可恢复的科学工作流重，系统究竟依据什么原则决定下一步如何执行”。

这个问题之所以重要，是因为蛋白质设计这样的科学工作流并不是简单的问答或单步工具调用任务。其执行过程通常包含多个强耦合阶段，不同步骤的成本和失败代价显著不对称，而且错误往往会沿工作流向后传递。一个系统即便具备了候选生成、结构预测、质量门控和人工确认能力，如果它仍然无法回答“现在是否应该继续计算、切换路径、请求人工接入或提前终止”，那么它本质上仍然只是一个更复杂的执行平台，而不是一个具备独立算法主线的研究系统。因此，在当前系统基础上，最自然的提升方向不是继续扩展设计能力，而是把研究对象提升为工作流运行时的治理控制问题。

也正是在这一层意义上，\texttt{controller}才成为比“继续做生命科学助手”更合适的研究主线。它使系统的核心问题从“完成一个生命科学任务”转变为“如何在风险、成本、恢复空间和人工预算约束下控制科学工作流的推进方式”。这不仅更接近控制、调度和序贯决策问题，也更有助于切断与“生命科学助手”父集之间的包含关系，因为\texttt{controller}所研究的是更底层的底层治理机制，而不是具体的知识问答或任务编排界面。

\subsection{为什么选择 Belief-State Controller}

在 \texttt{controller}的不同建模方式中，之所以特别考虑 \texttt{Belief-State Controller}，
是因为当前系统最关键、也最难被直接观测的因素，恰恰是都具有“隐状态”的特征。系统无法在执行之前明确知道某个任务到底有多难、当前候选是否已经足够可靠、此时的局部失败究竟是因为噪声还是结构性困难、剩余恢复空间是否足够支持继续自动执行、以及当前请求人工确认是否真正具有边际价值。换言之，系统面临的不是一个完全可观测的决策环境，而是一个必须根据运行时证据不断更新判断的不完全信息环境。

如果采用静态评分或规则门控的方式，这些问题通常会被简化为一组人工权重和阈值，例如“如果风险分高于某一阈值则请求HITL，否则继续执行”。这种做法虽然实现上简单，但会把“难度”退化为一个固定标签，把“控制”退化为启发式分支选择。这样形成的系统即使在工程上可用，也很难支撑一个有足够算法深度的毕业设计。相反，\texttt{Belief-State}的价值在于，它允许系统把“难度、风险、证据充分性、恢复余量和人工介入价值”统一看作运行时需要持续估计的\texttt{latent state}，并通过不断接收到的新观测对其进行更新。这样一来，难度不再只是一个离线打分结果，而成为一个真正进入决策闭环的状态变量。

更重要的是，\texttt{Belief-State}的引入，使控制问题具备了一个更清晰的算法表达。系统不再只是简单地根据当前观测做局部反应，而是维护关于当前执行状态\texttt{belief}，并再次基础上决定治理动作。这些治理动作不是普通的工具选择，而是工作流层面的决策，例如是否继续执行、是否扩大探索深度、是否切换到保守路径、是否进入局部\texttt{patch}、是否升级为\texttt{suffix replan}、是否请求人工确认、以及是否提前终止。也就是说，\texttt{Belief-State Controller}的核心不在于替代\texttt{Planner}或\texttt{Executor}，而在于作为它们之上的运行时决策层，对工作流的推进方式进行统一治理。这一点与当前系统已经具备的\texttt{FSM}、\texttt{HITL}和恢复链机制高度契合，也意味着\texttt{Belief-State Controller}既能利用现有系统作为实验平台，又不会退化为现有模块的简单重复。

\subsection{本文采用的Belief-State Controller}问题定义

在本课题中，\texttt{Belief-State Controller} 所面对的对象不是单轮对话，也不是单次模型推理，而是一条高代价科学工作流。该工作流可以被抽象为一个带有数据依赖与控制依赖的图结构，其中每个节点对应某一阶段的任务执行、质量判断、恢复操作或人工确认点。对于蛋白质设计场景而言，这样的图至少包含候选生成、结构预测、质量门控、结构条件优化以及恢复等关键部分。

在任一时刻，系统都处在一个无法被完全观测到运行状态之中。这个隐状态并不是单一的 \texttt{difficulty score}，而是一个结构化的状态集合，用于共同刻画当前任务的真实执行难度、当前证据是否充分、当前工具链是否稳定个、当前恢复空间是否仍然充足、以及当前是否值得引入人工干预。系统所能获得的只是运行过程中的局部观测。例如候选结果的一致性和分歧程度、结构预测输出及其质量标准、质量门控的拒绝原因、最近一次\texttt{patch}或\texttt{replan}的结果、已经消耗的时间与资源，以及此前人工决策所留下的轨迹。\texttt{controller}的任务，正是依据这些异构运行时观测，持续更新系统对于当前隐状态的\texttt{belief}，并据此在若干治理动作之间做出选择。

由此，本文中的\texttt{Belief-State Controller}可以被理解为一个面向科学工作流的部分可观测治理控制器。它不直接生成最终任务结果，而是以工作流的运行方式为控制对象，在成本、风险、时间和人工预算约束下，决定工作流下一步应如何推进、恢复、暂停或终止。这个定义具有两层意义：一方面，它使“难度”和“风险”真正进入系统决策闭环；另一方面，它也使课题从一个面向生命科学的助手系统，转化为一个面向高代价科学工作流的控制问题。

\subsection{相关研究现状及其启示}

围绕这一部分，现有研究已经提供了若干重要启示，但尚未直接给出与本课题完全重合的解决方案。

第一类相关工作是不确定性感知的agent规划。例如UoT\parencite{uot}提出在信息不足时让\texttt{language agent}根据不确定性主动进行信息搜索，这说明不确定性感知规划（uncertainty-aware planning）本身已经是被认可的研究方向，也证明“不确定性不是只能被动承受，而可以进入决策闭环”这一思路是成立的。与本课题相比，这类工作更多关注信息寻求与提问行为，而较少涉及高代价工作流中的阶段化恢复、人工介入和治理动作，因此为本研究提供的是方法论启发，而不是现成答案。

第二类相关工作是成本感知路由与\texttt{text-time}资源控制。BEST-Route\parencite{best-route}将问题难度（query-difficulty）与质量阈值纳入模型路由和采样预算分配，EcoAct\parencite{ecoact}则从经济性角度讨论\texttt{agent}系统中何时注册何种动作或工具。它们共同说明，在大模型系统中，根据难度和成本动态分配计算资源已经成为清晰的研究问题。这与本课题高度相关，因为\texttt{Belief-State Controller}同样需要根据状态估计决定是否继续投入更多执行决策。但不同之处在于，这些工作通常停留在模型路由或工具使用层面，而本文所关注的，是工作流层面的恢复、人工确认和执行治理。

第三类相关工作是长程价值引导与搜索控制。例如QLASS\parencite{qlass}通过逐步价值（stepwise value）引导agent在复杂任务中做更好的中间选择，Language Agent Tree Search\parencite{lats}则通过树搜索统一推理、行动与规划。这些工作表明，大模型agent的关键不只在于最终输出，更在于中间决策质量以及对未来收益的估计能力。对本课题而言，其重要意义在于说明：在长链路任务中，控制“下一步怎么走”本身就是研究重点。然而，这类任务仍更偏向搜索效率和推理路径优化，而不是面向高代价工作流的治理控制。

第四类相关工作是多Agent系统失败分析与结构化设计。Why Do Multi-Agent LLM Systems Fail?\parencite{whyfail}指出，多Agent系统的失败很大一部分并非来自模型本身，而来自角色失配、接口不清以及验证不足和终止条件设计不良。Which Agent Causes Task Failures and When?\parencite{whichandwhen}进一步说明，多Agent链式执行中的失败归因并不容易。于此同时，MetaAgent\parencite{metaagent}这类工作则显示，有限状态机等显式结构已经开始被用于更系统化地组织多Agent系统。这些研究共同提供了重要背景：在多Agent系统中，“如何治理执行过程”已经成为一个独立问题，而不是简单附属于任务求解过程。对本课题而言，这些工作一方面说明选择\texttt{controller}方向是合理的，另一方面也表明真正的难点不在于“增加一个模块”，而在于如何在带协作噪声和恢复链的环境中稳定做出治理决策。

综合这些文献可以看出，现有研究分别从不确定性、成本约束、价值引导、失败分析和结构化设计等角度接近了本课题，但尚未把以下几个因素统一起来：高代价科学工作流、显式恢复动作、HITL动作、结构化工作流状态，以及运行时\texttt{belief}更新。也正因如此，\texttt{Belief-State Controller}仍然具有明确的研究空间。它既不是凭空提出一个完全脱离文献背景的概念，也不是简单复用已有方法，而是在现有研究基础上，将问题收束到一个更贴近当前系统、也更贴近高代价科学工作流本质的问题上。

\subsection{为什么是Belief-State Controller}

因此，将当前选题进一步收束为\texttt{Belief-State Controller}，主要是因为它同时满足了三个条件。首先，它最能回应当前系统的真正缺口，即运行时控制依据的缺失。其次，它最能把“难度、风险、工作量、恢复与人工介入”统一为一个明确的算法问题，而不是若干零散机制的堆叠。最后，它在现有研究中有清晰的相关脉络可依托，又没有被已有工作完全覆盖，从而具备理论依据，也保留了可用于毕业设计的创新空间。

从系统角度看，当前已有的\texttt{FSM}、\texttt{HITL}、\texttt{patch/replan}和实验框架提供了\texttt{controller}的运行环境；从研究角度看，\texttt{Belief-State}为系统如何理解和更新“当前究竟处于什么状态”提供了更有力的算法表达。也正因如此，\texttt{Belief-State Controller}不只是一个可行方法，而是当前条件下最有可能同时兼顾算法性、独立性、系统可落地与时间可行性的方向。
